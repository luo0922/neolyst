# **分析师投研报告平台Overview**

## **1\. 概述**

### **1.1 核心目标：**

本平台的核心价值在于利用 AI 实现研报生产力的质变：

* **业务主导的自我进化**：通过自然语言对话沉淀业务规则（Skills）。Editor 和 SA 无需研发介入即可自行配置工作流、修改提取规则、优化审核逻辑，使系统能力随业务经验持续增长。  
* **全自动报告生产**：通过 AI 学习历史报告样本自动构建排版模板。系统自动解析草稿要素（含复杂图表、模型数据）并实现秒级排版，彻底消除人工调整格式的负担。  
* **合规风险自动拦截**：将合规检查（SFC 牌照、Stock List、利益披露、评级变动等）通过机器初筛与高亮提示，大幅降低人为疏漏风险。  
* **从“操作者”转向“监督者”**：通过渐进式自动化，目标是将 99% 的基础工作交给 AI。人工角色不再是排版和对数的工人，而是规则的制定者、生产过程的监督者与最终结果的确认者。

### **1.2 业务角色**

* **分析师 (Analyst)**：在微信里发送草稿和指令。后期直接与 AI Agent 协作。  
* **Editor**：在工作台处理任务。前期负责手动操作和教系统规则，后期负责维护系统和处理异常。  
* **审批/合规人员 (SA)**：在工作台查看系统初筛结果，完成审批和发布。  
* **系统管理员 (Admin)**：管理用户权限、行业分类等基础设置。  
* **外部投资者**：在外部发布系统查看报告。

## **2\. 平台架构**

各模块职责如下：

* **企业微信**：前端入口。负责收发文件和指令。  
* **Editor 工作台**：生产引擎。解析草稿、提取数据、自动排版，将结果写入 neolyst-db。  
* **SA 工作台**：审核引擎。执行合规检查，确认后修改 neolyst-db 状态为“已发布”。  
* **独立发布进程**：搬运工。看到报告状态变更为“已发布”，就自动把文件和数据抄写到 neoport-db。  
* **Neolyst-db**：内部数据库。存所有人员、权限、报告版本和操作日志。  
* **Neoport-db**：外部数据库。只存发给客户看的最终版报告。  
* **Web 展示层**：查询网页。内部用 Neolyst-web 查审计日志，外部用 Neoport-web 看报告。

## **3\. 核心机制**

### **3.1 权限管控与审计分离**

* **实名登录与写库**：所有人员（Editor/SA）均使用专属实名账号登录工作台与底层数据库交互。  
* **代建放行与归属解耦**：底层安全策略允许 Editor 代理分析师创建报告。此机制既确保了审计日志记录“真实操作人 (Editor)”，又保障了业务上“报告归属人 (Analyst)”的准确。

### **3.2 渐进式自动化**

* **人机协作**：系统自动处理标准提取、排版与合规工作；系统无法处理的特例由人工兜底。  
* **经验固化**：人工在兜底处理后，可通过自然语言对话将经验配置给系统，转化为新的自动化规则（Skills）。

## **4\. 核心工作流**

### **4.1 报告生产**

1. **任务接收**：分析师通过微信发送草稿与指令。  
2. **AI 预处理**：Editor 登录工作台（初期为 opencode），指定归属分析师，调用自动化技能。  
3. **确认入库**：系统解析并排版。Editor 核对后确认，系统以 Editor 身份写入 neolyst-db，状态标记为“待审核”。

### **4.2 审批发布**

1. **自动初筛**：SA 登录工作台调用审核功能，系统高亮合规风险点。  
2. **人工决断**：SA 确认后下达发布指令。  
3. **异步发布**：系统修改 neolyst-db 状态为“已发布”，独立进程侦测后自动同步至 neoport-db。

## **5\. 代码库划分**

* **neolyst**：内部管理系统仓库。含 neolyst-web 代码及 neolyst-db 的物理模型。  
* **neoport**：外部发布系统仓库。含 neoport-web、neoport-db 及 Publish Worker 进程。  
* **neoskills**：业务逻辑库。封装解析、要素提取、排版渲染、模板自动生成、合规检查等所有原子能力。  
* **neospace**：工作台仓库。Skills 的运行载体，负责编排具体的业务流程。

## **6\. 技术栈总览**

* **数据与权限**：Supabase (PostgreSQL \+ RLS 安全策略 \+ Storage)。  
* **前端框架**：Next.js (App Router)。  
* **业务逻辑层**：Python (核心文档处理、数据解析、图表生成)。  
* **部署/运行**：  
  * 初期：利用 **opencode** 挂载 Github 仓库作为临时工作台。  
  * 后期：部署自研容器化工作台。

## **7\. 实施规划**

### **7.1 第一阶段：业务启动 (MVP)**

* **目标**：解决排版工作量最大的问题。  
* **做法**：用 opencode 挂载技能库，由 Editor 手动提交原始报告进行解析排版。

### **7.2 第二阶段：工作台成型**

* **目标**：自研工作台上架，覆盖翻译、刷新、合规等全部自动化技能。  
* **做法**：上线 neospace，Editor 继续负责规则固化和业务流程串联。

### **7.3 第三阶段：分析师直连**

* **目标**：**分析师直连，自动干活**。  
* **做法**：系统直接对接微信。分析师发报告，系统自动处理并推给 SA。标准报告不再需要 Editor 介入，Editor 转型为系统监督者。
